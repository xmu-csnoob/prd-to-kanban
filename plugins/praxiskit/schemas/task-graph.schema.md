# Task Graph Schema v3.0

The `task_graph` artifact is a directed graph of typed tasks: each task has an ID, dependencies, write scope, and acceptance criteria. The same artifact serves both light (`seed-to-task-graph`) and standard/heavy (`prd-to-task-graph`) recipes.

This schema is used for **pre-flight validation** by the producing transform — it does not trigger a clarification gate. The transform must refuse to proceed if blocking conditions are not met.

## Pre-flight Blocking Conditions (PRD-sourced)

For `prd-to-task-graph`, run these checks before decomposing tasks. Any failure → refuse and output remediation steps.

| Condition | Check | Remediation |
|---|---|---|
| No blocking Open Questions in PRD | Scan `## Open Questions` table for rows where `Blocks` = `implementation` or `release` and status is unresolved | List the blocking questions; instruct user to resolve them via `idea-to-prd` or the clarification gate |
| All FRs have acceptance criteria | Check every FR row for non-empty `Acceptance Criteria` | List FRs without criteria; route back to `idea-to-prd` |
| At least 1 milestone defined | Check `## Milestones` section | Instruct user to define milestones in PRD |

## Pre-flight Blocking Conditions (seed-sourced)

For `seed-to-task-graph` (light recipe), run these checks before decomposing tasks:

| Condition | Check | Remediation |
|---|---|---|
| Seed has confirmed intent | Seed contains a clear goal statement OR the gate collected one | Fire `seed.schema.md` gate to capture intent |
| At least 1 acceptance signal | User provided observable success criteria via seed or gate | Fire gate with question "How will you know this is done?" |
| Domain-specific lists captured | Cross-field rules from `idea.schema.md` apply (e.g., cache metadata) | Fire gate with the matching `forbidden-to-infer` field |

## Task Field Rules

Each task row has these required fields:

| Field | State | Notes |
|---|---|---|
| task_id | inferable | Sequential `T{wave}.{n}` format |
| title | inferable | Derive from FR, milestone, or seed intent |
| acceptance_criteria | filled-by-user | For PRD-sourced: copy verbatim from FR row. For seed-sourced: copy, quote, or split the seed/gate answer across subtasks without adding new acceptance meaning. |
| write_scope | inferable | Files/dirs this task will modify |
| dependencies | inferable | Other task IDs this task depends on |
| status | inferable | `[ ]` pending, `[/]` in progress, `[x]` done, `[!]` blocked |

## What to Do With Non-Blocking Open Questions

Non-blocking Open Questions in the PRD may proceed to `task_graph`. Add a corresponding task:

```
| T{n} | Resolve: {question summary} | [ ] | Owner: User | — | [{dependent task IDs}] |
```

## Output Format (work/task-graph.md)

```markdown
# Task Graph: {project}

## Source
- {seed | PRD} → `work/{file}`
- Generated by: `{transform-name}` on {date}

## Overview
- Total tasks: N
- Waves: 0..K
- Critical path: {T_ID, T_ID, ...}

## Tasks

| ID | Title | Status | Acceptance Criteria | Write Scope | Dependencies |
|----|-------|--------|---------------------|-------------|--------------|
| T0.1 | ... | [ ] | Given..., when..., then... | `path/` | — |
| T1.1 | ... | [ ] | ... | `path/` | [T0.1] |

## Dependency Graph
{textual or mermaid diagram showing wave layering}

## Parallelism Windows
- Wave 1: T1.1 ‖ T1.2 (disjoint write scopes)
- Wave 2: T2.1 (sequential)
```

## Migration from v2-M kanban

v2-M projects with `work/kanban.md` migrate by renaming the file to `work/task-graph.md`. No content changes required if the v2-M kanban followed the same task field structure.
